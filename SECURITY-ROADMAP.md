# Clay Incoming-Connection Hardening — Implementation Roadmap

**Status: implemented (Phases 0–5), 2026-07-13. D6 (ban-exemption bugfix) added
2026-07-13.** This is now a design record, not a TODO list. The wire protocol (D4), the
routing matrix (D2), the gate rules (D3), and the ban-exemption rule (D6) are the
authoritative spec for the code in `src/http.rs`. User-facing summary of the behavior
change: `SECURITY-NOTES.md`.

This document is self-contained: each phase can be implemented without conversation
context. All file:line references were verified against the codebase when this was
written (July 2026); line numbers may drift — anchor on the named functions.

Build/verify after every phase:

```bash
cargo build --target x86_64-unknown-linux-musl --no-default-features --features rustls-backend
cargo test --no-default-features --features rustls-backend
```

Also compile-check the native-tls and plain variants when touching src/http.rs — all
three server paths are affected. NEVER use `println!`/`eprintln!`/`dbg!` (corrupts the
TUI); use `debug_log()` / `log_remote_event()`.

## Goal

Today Clay's embedded HTTP+WebSocket server (one listener, default `0.0.0.0:9000`)
serves the full web UI to any IP with no authentication — only the WebSocket layer
requires a password or auth key. Scanners see a valid page, the plain-HTTP→HTTPS 301
redirect confirms the server exists, and the allow list only gates password auth, not
connections. After this roadmap:

| Scenario | Behavior |
|---|---|
| No allow list | Web UI served to any IP but only under `/clay`; WS auth accepts **password OR auth key** (CHANGES current behavior — today an empty allow list rejects password auth) |
| Allow list set, IP matches | May fetch `/clay` UI; must still authenticate (password or auth key) — list membership never skips auth |
| Allow list set, IP doesn't match | **Every** connection silently dropped (no TLS handshake, no redirect, zero bytes). Only entry: auth-key knock → WebSocket only; web UI stays locked |
| Android + auth key | Always allowed (knock when not allow-listed; direct WS otherwise) |
| Android + password | Only when no allow list, or IP matches allow list |
| Random probes | Silent drop; invalid paths record a strike under the existing 2-strike ban; `/favicon.ico`/apple-touch-icon exempt from strikes |
| Localhost | Always fully allowed, zero config (GUI WebView depends on `127.0.0.1`) |

## Current architecture (reference — do not re-derive)

- Unified server: one TCP listener on `0.0.0.0:{http_port}` serves static HTTP and WS.
  `route_connection()` (src/http.rs:349) dispatches WS upgrade vs static HTTP →
  `handle_http_routes` (src/http.rs:265). Three variants: native-tls
  (`start_https_server` src/http.rs:489), rustls (src/http.rs:639), plain
  (`start_http_server` src/http.rs:979). TLS variants peek the first byte
  (src/http.rs:570/752): `0x16` = TLS ClientHello, else 301-redirect to HTTPS via
  `redirect_http_to_https` (src/http.rs:143).
- Static routes (GET only, no auth): `/`, `/index.html`, `/style.css`, `/app.js`,
  `/theme-editor`, `/keybind-editor`, `/action-editor`, `/fonts/*.woff2`,
  `/favicon.ico` (204). Unknown → 404 + `log_http_404` + `ban_list.record_violation`
  (src/http.rs:428-432). Non-GET → 405 (src/http.rs:275). Assets embedded via
  `include_str!`/`include_bytes!` (src/http.rs:111-130).
- `BanList` (src/http.rs:802-887): in-memory, 2 violations → ban, localhost never
  banned, not persisted. Banned IPs are skipped at accept (src/http.rs:552).
  Connection caps: `MAX_HTTP_CONNECTIONS_PER_IP=10`, `MAX_WS_CONNECTIONS_PER_IP=20`.
- WS auth (`handle_ws_client`, src/websocket.rs:1383-1712): server sends
  `ServerHello { multiuser_mode, challenge }` (random challenge). 30s auth timeout.
  Client sends `AuthRequest { password_hash?, username?, auth_key?, request_key,
  challenge_response }`. Auth-key path forwarded to app via
  `AppEvent::WsAuthKeyValidation` → `handle_ws_auth_key_validation`
  (src/main.rs:6248-6307); challenge-response = SHA256(stored_key_hex + challenge) as
  hex strings (`hash_with_challenge`, src/websocket.rs:1072). Password path inline
  (src/websocket.rs:1591-1625). Current allow-list enforcement for password auth:
  rejected if allow list EMPTY ("auth key required", src/websocket.rs:1564-1574);
  rejected+violation if IP not listed (src/websocket.rs:1575-1586). Failed auth-key
  deliberately records NO violation (src/main.rs:6293-6298).
- Allow list: `websocket_allow_list` CSV setting; `is_in_allow_list`
  (src/websocket.rs:1106): `*`, IP wildcards (`192.168.1.*`), exact IP, hostname
  patterns via `reverse_dns_lookup` (src/websocket.rs:1148) when
  `is_hostname_pattern` (src/websocket.rs:1081). Runtime `whitelisted_host`
  (single IP auto-auth after successful login, never persisted).
- Auth key: single `websocket_auth_key: Option<AuthKey>` (src/main.rs:1400,1451),
  generated by `App::generate_auth_key` (src/main.rs:6325), AES-256-GCM encrypted in
  `~/.clay/settings.dat` (src/persistence.rs:214-216, load ~734-753, reload ~2263).
  Regen/revoke: `handle_ws_key_request`/`handle_ws_key_revoke` (src/main.rs:6309-6322)
  and the TUI web popup.
- Android: WebView loads UI from bundled APK assets
  (`file:///android_asset/web/index.html`, MainActivity.java:883) — only the WS goes
  over the network, via OkHttp in NativeWebSocket.java (`connect(String url)` at :46,
  OkHttpClient built at :76 with a custom `SSLSocketFactory`). Auth key stored in
  SharedPreferences (MainActivity.java:239-253); app.js presents it as
  SHA256(key+challenge) (`tryAuthWithKey`, src/web/app.js:2945).
- remote.log: `log_remote_event(event, ip, details)` (src/http.rs:66-83).
- Multiuser mode (`--multiuser`): per-user passwords from multiuser.dat, auth keys
  disabled, same web-server settings.
- GUI mode (Windows/macOS default): local WebView loads `http://127.0.0.1:{port}/` —
  localhost must always work with zero config.
- Repo rule (CLAUDE.md): every new setting must appear in ALL THREE interfaces
  (console TUI popup, web, webview-GUI) and be saved/loaded/reloaded in persistence.rs.

## Design decisions (locked — do not re-derive)

### D1. One new setting: `web_path` (String, default `"clay"`)
- Non-empty (default): stealth mode — UI served only under `/{web_path}/...`, every
  other path silently dropped.
- Empty string: legacy mode — UI at `/`, unknown paths get today's 404 + violation.
  This is the single backward-compat escape hatch.
- The IP gate (D3) and knock (D4) are NOT behind a setting — they activate
  automatically when `websocket_allow_list` is non-empty / when a knock arrives.
- **Compatibility break (release notes):** bookmarks to `/` stop working for
  non-localhost users; old Android APKs (no knock) only work from allow-listed IPs.
  Remedy: set `web_path` empty, or add the device IP to the allow list.

### D2. Path routing rules (in shared `decide_route`/`handle_http_routes`)
Let `P = web_path`. Decided per-request after the IP gate:
- **Localhost:** routes served at BOTH legacy roots (`/`, `/style.css`, …) AND
  `/{P}/...` — the GUI WebView must keep working unchanged.
- **Allowed non-localhost** (allow-listed, whitelisted_host, or open mode):
  - `GET /{P}` → 301 to `/{P}/` (index.html uses relative hrefs: src/web/index.html:7,933).
  - `/{P}/` and `/{P}/index.html` → index; `/{P}/style.css`, `/{P}/app.js`,
    `/{P}/theme-editor`, `/{P}/keybind-editor`, `/{P}/action-editor`,
    `/{P}/fonts/*.woff2`, `/{P}/favicon.ico` (204) → existing responses.
    Implementation: strip the `/{P}` prefix, map `""`→`"/"`, feed the remainder into
    the existing `match path` block (src/http.rs:286).
  - Any other path → **silent drop** (zero response bytes, close TCP) + violation,
    EXCEPT `/favicon.ico` and paths starting `/apple-touch-icon` (still dropped, NO
    strike — browsers auto-request them; striking would self-ban legit users).
    `/robots.txt`, `/.well-known/*`, `/` DO strike.
  - Grace: an IP **in the allow list** (not merely open mode) requesting `/` or
    `/index.html` gets 301 → `/{P}/`. Open-mode clients get the silent drop at `/`.
  - Non-GET methods: in stealth mode, silent drop + violation (replaces the 405).
- **Knocked connections** (D4): any static HTTP request → silent drop, **no
  violation** (key already proven; don't self-ban a buggy client), log
  `KNOCK-HTTP-DENIED`.
- **WebSocket upgrade** (branch at src/http.rs:379): accept at ANY path if knocked OR
  localhost OR allow-listed/whitelisted (keeps old Android APKs working for
  allow-listed users). Open mode, non-localhost: path must be exactly `/{P}/ws` (any
  path if `P` empty); wrong path → silent drop, **no violation** (updated Android
  probes `/clay/ws` then `/ws`; striking would ban legitimate devices), log
  `WS-PATH-DROP`.

### D3. IP gate (accept time, all three server variants)
When `websocket_allow_list` is non-empty: a non-localhost, non-listed,
non-whitelisted IP gets **every** connection dropped with zero bytes before the TLS
peek/handshake — no 301, no certificate — unless it performs a successful knock.
- Reuse `is_in_allow_list`/`reverse_dns_lookup`/`is_hostname_pattern` from
  websocket.rs. Reverse-DNS only when hostname patterns are present in the list.
- Gate drops log `GATE-DROP` and record **NO ban violation**. Rationale: banning a
  non-listed IP would also block its knock (ban check runs before accept),
  permanently locking out the one legitimate entry path (e.g. an old Android app that
  retries twice before the user updates it). Violations still come from: bad knock
  magic, failed knock digest, stealth path probes, oversized requests, tls-timeout,
  failed WS password auth.
- The 301 HTTPS redirect (`redirect_http_to_https`) fires only for connections that
  passed the gate AND whose request path starts with `/{P}` (or `P` empty, or
  connection knocked); otherwise silent drop with the same favicon exemption.

### D4. Knock wire protocol — "CLAY-KNOCK v1"
Binary, fixed-size, request/response, before TLS/HTTP, on the same socket. 10s
timeout per read (reuse `READ_TIMEOUT_SECS`).

1. **C→S HELLO (6 bytes):** `C7 4C 41 59 01 00` — magic `0xC7 'L' 'A' 'Y'`, version
   `0x01`, reserved `0x00`. First byte `0xC7` is: not `0x16` (TLS ClientHello), not
   ASCII (no HTTP method collision), and an invalid UTF-8 lead byte. The existing
   1-byte peek disambiguates all three protocols: `0xC7`=knock, `0x16`=TLS, else HTTP.
2. **S→C CHALLENGE (34 bytes):** `C7 4B` + 32 random bytes (`getrandom`).
3. **C→S RESPONSE (32 bytes):** raw `SHA256(auth_key_utf8_bytes || challenge_bytes)`.
   Raw-bytes digest — NOT the hex-string convention used by `hash_with_challenge`;
   put a comment next to both noting the difference.
4. **Server verifies** with a constant-time compare. Success → **S→C ACK (2 bytes):**
   `C7 06`; mark connection `knocked=true`; log `KNOCK-OK`;
   `ban_list.clear_violations(ip)`. Failure (wrong digest, no stored key/multiuser,
   short read, timeout) → log `KNOCK-FAIL` (or `KNOCK-BAD-MAGIC`),
   `record_violation`, drop with zero further bytes.
5. Client immediately continues on the same socket: TLS ClientHello (wss) or HTTP WS
   upgrade (ws). All knock reads are `read_exact` of fixed sizes, so the stream is
   positioned exactly at the next protocol byte.

Replay-proof: fresh 32-byte challenge per connection. The knock is **transport gating
only** — it does NOT pre-authenticate the WS session; normal WS auth (auth key again)
still follows. Chosen because it's simpler (no auth state plumbed through TLS accept
into `handle_ws_client`) and safer (two independent proofs); the client already does
key-based WS auth, so the second proof costs nothing. Knocks are accepted from ANY
non-banned IP including allow-listed ones (the client can't know its list status).

Server needs the plaintext auth key synchronously at accept time: share via
`Arc<std::sync::RwLock<Option<String>>>` (see `SecurityGate`, Phase 0), updated by
`handle_ws_key_request`/`handle_ws_key_revoke` and settings load/reload. In multiuser
mode the Arc holds `None` → every knock fails (documented limitation: non-allow-listed
hosts cannot connect at all in multiuser).

### D5. WS auth-rule change
In `handle_ws_client`: delete the `allow_list_empty` rejection block
(src/websocket.rs:1564-1574) and the `allow_list_empty` computation (~1429-1432).
New matrix: empty allow list → password auth allowed from anywhere (if
`password_enabled`); non-empty list → password only from localhost/listed/whitelisted
(existing block at 1575-1586 stays, including its violation). Auth-key WS path
unchanged.

### D6. Ban-exemption bugfix — allow-listed IPs are never banned for probes (added
2026-07-13, fixing a bug found in testing D2/D3)

**The bug:** with `web_secure=true` and an allow list configured, an allow-listed user
hitting `https://host:9000/` got the correct D2 grace redirect (301 to `/{web_path}/`,
no strike). The same user hitting `http://host:9000/` (omitting the "s") got banned
after two tries. Root cause: `redirect_http_to_https()` — the handler for a plain-HTTP
request arriving on the HTTPS port — re-derived path reachability with its own
`path_allowed` check instead of calling `decide_route()`, and that check never received
`in_allow_list`. The duplicate logic drifted out of sync with D2 the moment D2 grew the
allow-listed grace redirect. A second, related hole: `route_connection`'s
unparseable-request branch was a bare `None => return` — no log, no strike either way —
so a TLS ClientHello sent to a plain-HTTP server (browsers do this on their own from a
cached HSTS/HTTPS-First decision) was invisible in `remote.log`.

**The deeper realization, generalized:** once `websocket_allow_list` is non-empty, D3's
accept-time gate drops every non-listed, non-localhost, non-knocked IP before it can
reach *any* probe-strike site in this file. A scanner therefore can never trigger a
probe strike at all — the ban counter, as implemented pre-D6, was pointed exclusively at
its own legitimate, allow-listed users (path probes, knock failures, TLS timeouts,
oversized requests, malformed requests all fed the same 2-strike counter that could only
ever catch someone who was supposed to be let in).

**Decisions:**
1. **Allow-listed IPs (and localhost, as before) are never banned for probes** — path
   probes, stale bookmarks, http/https typos, knock failures, TLS timeouts, malformed
   requests. They are still silently dropped (they learn nothing from the network), just
   never banned. Implemented as one chokepoint, `SecurityGate::strike()`, which every
   probe-strike site in `src/http.rs` now calls instead of `ban_list.record_violation()`
   directly. `strike()` consults `is_ban_exempt()`, which matches localhost, the runtime
   `whitelisted_host`, or a *specific* allow-list entry (exact IP, IP wildcard, or
   hostname pattern via reverse DNS, same resolution `gate_connection` already does). A
   bare `*` allow-list entry is deliberately excluded from exemption: an allow list of
   `*` means "let everyone reach the UI," not "nobody on the internet can ever be
   banned" — a completely open server should keep its scanner-ban protection.
2. **Failed WebSocket password auth still bans — threshold raised from 2 to 5**, for
   everyone, allow-listed or not. This is the one strike on this path that actually
   protects something (the password itself), so it's exempt from the D6 allow-list
   carve-out — but 5 gives room for a typo without inviting unlimited brute-force from a
   LAN device. Implemented as `BanList::record_auth_failure()`, sharing
   `record_violation`'s bookkeeping (same per-IP counter, same localhost exemption) via
   a shared `record_with_threshold(ip, reason, threshold)` helper, just with `threshold
   = 5` instead of `2`.
3. **A TLS ClientHello arriving at a plain-HTTP server is logged, never struck.**
   Browsers send these on their own (HSTS cache from a previous `web_secure=true` run,
   HTTPS-First mode) — striking it would ban users for something they didn't do, the
   same rationale as the existing favicon exemption. Logged as `TLS-ON-PLAIN`.
   `route_connection`'s unparseable-request branch now distinguishes: first byte `0x16`
   → `TLS-ON-PLAIN`, no strike; anything else unparseable → `HTTP-DROP` + strike (it's
   genuine garbage, not a browser behaving normally).
4. **A knocked WebSocket connection that fumbles the password isn't struck for "not in
   allow list" either.** With D3's gate active, the *only* way a non-listed IP can reach
   that rejection branch in `handle_ws_client` at all is a successful knock — proof it
   holds a currently-valid auth key (a revoked key fails the knock). If such a device
   then falls back to a password prompt (e.g. the key was rotated on the phone) and gets
   it wrong twice, the pre-D6 code banned it — and a banned IP can't even knock (the ban
   check runs before accept), so the device would be locked out of its own recovery path
   until the server restarts. `knocked: bool` is now threaded into `handle_ws_client`
   (one call site, `route_connection`) specifically to skip that strike; the rejection
   itself is unchanged (still refused).
5. **`redirect_http_to_https()` now calls `decide_route()` directly** instead of
   maintaining a second reachability check — this is the actual fix for the reported
   bug, not merely the exemption rule. Its signature changed to take `gate:
   &SecurityGate` and `in_allow_list: bool` (both TLS accept loops already had
   `in_allow_list` in scope from `GateResult::Proceed`, so this was a threading fix, not
   new plumbing). `decide_route`'s `Serve`/`Redirect`/legacy outcomes all mean
   "reachable" → 301 to the HTTPS equivalent (using `decide_route`'s already-resolved
   `Redirect` location so the grace case reaches `/{web_path}/` in one hop, not two);
   `SilentDrop` means drop, striking only if `violation.is_some()` (and the IP isn't
   exempt). Because it now reuses the exact same decision function `route_connection`
   uses for every other HTTP(S) request, these two call paths cannot drift apart again.
6. **Test isolation:** `cargo test` was appending real `BANNED`/`KNOCK-*`/etc. lines to
   the developer's live `~/.clay/remote.log` — the test suite exercises the same gate/
   ban/knock code paths production does. `log_remote_event()` is now a no-op under
   `#[cfg(test)]` (a separate zero-argument-use function, not a runtime early-return —
   avoids an `unreachable_code` warning on the real implementation).

**Not changed:** the D3 gate-drop rule (no violation on `GATE-DROP`, unrelated to
strikes since it isn't even a strike site), the favicon/apple-touch-icon exemption (D2,
orthogonal — it exempts by *path*, D6 exempts by *caller*), and the WS-PATH-DROP /
KNOCK-HTTP-DENIED no-strike rules (D2/D4, already correct — this bug was specifically
about the two places that used to disagree with `decide_route`/D3's threat model, not
about the no-strike rules that were already right).

### D7. Security-audit remediation (added 2026-07-14)

A follow-up audit (three parallel reviews, every finding verified against the code)
surfaced weaknesses across the web frontend, multiuser mode, client-side TLS, file
permissions, and resource limits — all *outside* the incoming-connection surface D1–D6
hardened. Fixed in three severity-ordered phases. The load-bearing decisions:

**Client-side TLS is now trust-on-first-use (TOFU), not accept-all.** Every outbound TLS
client path (remote-console channel `remote_client.rs`, WebView proxy `webview_gui.rs`,
MUD world connections and the hot-reload TLS proxy in `platform.rs`, `/connect` in
`commands.rs`, multiuser/daemon world connects in `daemon.rs`) previously used
`NoCertificateVerification`, which asserted *both* the cert chain and the handshake
signature unconditionally — pure MITM exposure on the channel that carries the
auth-key/password handshake and full command injection. Replaced with `TofuVerifier`
(`platform::danger_rustls`): SHA-256 the presented end-entity cert DER, key on the SNI
host, and pin it in `~/.clay/known_hosts.dat` (0600) on first sight — **silent, no
first-connect prompt**. A later mismatch is *blocked* and surfaced in all three UIs with
old vs new fingerprint and an explicit "trust new certificate" action (`replace_pin` +
reconnect). CA verification was deliberately *not* used: Clay's own `web_secure` server
and most hobbyist MUDs present self-signed certs, which CA verification would reject.
**Critical correctness point:** `verify_tls12_signature`/`verify_tls13_signature`
delegate to the real `rustls::crypto` verifiers (default `ring` provider), NOT
`assertion()`. Certificates are public and sent in the clear during the handshake;
without proving private-key possession, an attacker could replay the pinned cert and
defeat the pin. The native-tls MUD path can't reject *during* the handshake (no
pluggable verifier) so it checks the peer-cert fingerprint post-handshake
(`platform::check_native_tls_peer_pin`) — native-tls still does its own signature
verification internally, so the replay defense holds there too. A regression test drives
a real rustls handshake through pin → mismatch-blocked → *pinned-cert-replayed-with-
wrong-key-still-fails*, the last case proving rejection came from signature verification,
not the fingerprint.

**Web-client XSS from MUD output (`app.js`).** `escapeHtml` didn't escape `"`, and
`convertDiscordEmojis` re-injected an unescaped `[^:]+` name capture into an `<img
alt="…">` — so a crafted MUD line broke out and landed an `onerror` handler, running by
default (`showTags` defaults false). It pivoted to auth-key theft and, via `/load`/
`/save`, host file I/O. Fixed at three layers: `escapeHtml` now escapes `"`/`'`;
`convertDiscordEmojis` restricts the name to `[A-Za-z0-9_]+` and escapes what it
interpolates; and `sanitizeHtml` now wraps the scrollback/new-line sinks, not just the
prompt.

**Multiuser isolation.** `ConnectWorld` had no ownership check and auto-replayed the
*owner's* stored `connect <user> <password>` — any authenticated user could seize
another's character with no password. Now gated on `world.owner == username`, matching
the sibling handlers. `build_multiuser_initial_state` also stopped broadcasting every
world's hostname/port/MUD-login to every user (now redacted for non-owners; passwords
were already blanked). And `ChangePassword` stored the client-sent hash into the
plaintext field, which got re-hashed at load into `SHA256(SHA256(pw))` — a permanent
lockout; fixed with a `password_is_hash` marker so the loader/installer treats an
already-hashed credential as final.

**Defense-in-depth / hygiene.** WS frames are now capped (256 KB message / 64 KB frame)
via `accept_async_with_config`, closing pre-auth memory amplification. All at-rest secret
files (`secure.key`, `settings.dat`, `multiuser.dat`, `key.pem`, the reload-state file,
`settings-audit.log`) are created 0600 and `~/.clay` is 0700, via
`util::write_secret_file`/`secure_create_file`/`secure_append_file` (the TLS private key
`key.pem` was previously world-readable). Static-secret comparisons use the shared
`util::constant_time_eq`. The GMCP media auto-download validates the URL (http/https
only; `file:`/loopback/link-local/RFC1918 rejected) and constrains curl with `--proto`/
`--proto-redir` — closing a MUD-triggerable SSRF + local-file read. RNG failures fail
closed (WS challenge refuses the connection; `generate_auth_key` returns `None`) instead
of proceeding with zeroed bytes. The plain-HTTP→HTTPS redirect sanitizes the reflected
`Host` header. `println!`/`eprintln!` were routed through `debug_log` per the TUI rule.

**Flagged, not fixed (needs its own cycle):** `cargo audit` reported vulnerabilities in
`rustls-webpki` (0.101/0.102), `idna` (0.5, behind the `url=2.5.0` pin), and `time`
(<0.3.37 pin) — all requiring a rustls major bump and/or a Rust-toolchain move off the
1.75 pins, so they were reported rather than forced. Clay's real exposure to the
`rustls-webpki` name-constraint/CRL bugs is low: `TofuVerifier` pins fingerprints and
uses the crypto provider's signature verification directly rather than exercising
webpki's cert-chain/name-constraint/CRL path, and the server uses `no_client_auth`.

**Not changed (documented tradeoffs):** unsalted SHA-256 password hashing (the hash is a
bearer credential by design for the JS client), `LEGACY_ENCRYPTION_KEY` (kept as a
decrypt-only fallback, now logged when it fires so stale blobs get re-encrypted), and
`with_no_client_auth()` on Clay's own server (correct — Clay authenticates by
password/knock, not client certs).

### D8. Always-secure web server, auto-cert in settings.dat, localhost always plain
(added 2026-07-20)

**Goal:** remove the Protocol (Secure/Non-Secure) choice from `/web` — new users
shouldn't have to understand TLS to be reachable remotely. The server is now always
TLS-capable for remote clients; `web_secure` is no longer read to select HTTP vs HTTPS
(kept in `Settings`/`GlobalSettingsMsg`/settings.dat only for wire/import compat — always
sent as `true`, never meaningfully toggled by any UI).

**Localhost is always plain, remote is always TLS on the same port.** In the HTTPS accept
loop (`http.rs`, both native-tls and rustls variants), the existing first-byte peek that
disambiguates a plain HTTP request from a TLS ClientHello now branches on
`addr.ip().is_loopback()`: a loopback connection is served plain via `route_connection`
(no cert, no handshake — same call `start_http_server` already makes), everything else
still gets the existing redirect-to-HTTPS. This means the desktop GUI WebView (which
talks to `ws://127.0.0.1`) never sees a certificate prompt, while a remote browser or
Clay instance on the same port always gets TLS. `start_http_server`/plain-HTTP-only mode
is no longer used by the single-user startup paths or `restart_http_server` — only by the
`--multiuser` daemon (`daemon.rs`), which is untouched by this change and keeps its own
`web_secure` toggle and manual cert config; it isn't reachable from the `/web` popup.

**Auto-generated cert is now canonically stored in settings.dat, not just on disk.**
`resolve_web_cert_files()` (main.rs) resolves, in order: (1) a user-provided cert
(`websocket_cert_file`/`websocket_key_file`, unchanged from D7 — "Custom Cert File: Yes"
in the popup), else (2) the auto-generated cert, whose PEM content now lives in two new
encrypted-settings fields, `web_cert_pem` (cleartext, base64'd only to stay on one INI
line — it's public) and `web_key_pem` (base64'd then run through the existing
`secret()`/`encrypt_password()` at-rest encryption, same mechanism as
`websocket_auth_key`/world passwords). The resolver materializes/refreshes
`~/.clay/cert.pem`/`key.pem` on disk (a derived cache the file-based TLS server APIs
still read) and re-syncs settings.dat whenever `generate_self_signed_cert()` runs. Net
effect: the private key never leaves this machine's `secure.key`-encrypted blob, and if
`settings.dat` is `/import`ed to another machine the key silently fails to decrypt there
and gets regenerated rather than leaking across machines.

**Auto-approval on the Clay↔Clay path was already solved — no new work needed.** The
D7 `TofuVerifier`/`known_hosts.dat` pinning already gives every Clay-to-Clay connection
(remote console, another instance's WebView proxy, the Android app) silent trust-on-first-
connect and a blocking cert-mismatch prompt only if the fingerprint later changes. D8
only had to make the *server* side always present a cert for that to kick in on every
remote connection, not just when a user had manually flipped Protocol to Secure before.
A plain browser (which does full CA validation, not TOFU) still sees a one-time
self-signed-cert warning unless the user supplies a CA-signed cert via Custom Cert File.

**GlobalSettingsMsg never carries the private key.** `web_cert_pem`/`web_key_pem` were
deliberately *not* added to the WS wire struct; only the existing `tls_configured`
boolean (now meaning "a custom/user cert is configured") and `ws_cert_file`/`ws_key_file`
(user-provided paths only — empty when using the auto-cert) travel over the wire, matching
the pre-existing "server never sends the auth key/password to unauthenticated clients"
posture.

**Not changed:** the pre-existing "remote WS clients can set but not clear
`ws_cert_file`/`ws_key_file` via `UpdateGlobalSettings`" asymmetry (`main.rs`/`daemon.rs`
only apply that message's cert fields when non-empty, by original design — "remote clients
send empty for security"). Toggling Custom Cert File from Yes back to No is fully correct
through the popups' own Save path (`apply_web_settings`, unconditional) in all three UIs;
only the narrower case of clearing it from a genuinely separate remote console session
inherits this older limitation, unrelated to D8.

2. NativeWebSocket: add `connect(String url, String authKey)`. When the key is
   non-empty, build the OkHttpClient with `.socketFactory(new KnockSocketFactory(authKey))`:
   - `KnockSocketFactory extends javax.net.SocketFactory`, returns
     `new KnockSocket(authKey)` from every `createSocket()` overload.
   - `KnockSocket extends java.net.Socket`, overrides `connect(SocketAddress, int)`:
     `super.connect(...)`, `setSoTimeout(10000)`, then over raw streams: write the
     6-byte HELLO; `readFully` 34 bytes; verify `C7 4B`;
     `MessageDigest.getInstance("SHA-256")` over `key.getBytes(UTF_8)` + challenge;
     write the 32-byte digest; `readFully` 2 bytes; verify `C7 06`; restore
     `setSoTimeout(0)`. Any mismatch/timeout → `throw new IOException("knock rejected")`.
   - OkHttp creates the raw socket via `socketFactory`, connects it (knock runs), and
     only then layers `sslSocketFactory` (wss) or writes the HTTP upgrade (ws) —
     exactly the required ordering. Each pooled/retried connection knocks
     independently (fine; the server treats every TCP connection separately).

   WITHOUT the knock factory (runtime per-session flag). Old servers never answer
   `C7 4B`, so detection is reliable.
